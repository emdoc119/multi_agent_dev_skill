# SKILL_multi_agent_dev_v2.md

## 목적
이 skill은 다양한 소프트웨어 프로젝트에 재사용 가능한 **멀티 에이전트 개발 파이프라인**을 정의한다. 목표는 단일 코딩 프롬프트가 아니라, PM-Collector-Architect-Coder-Reviewer-QA Gate의 협업 루프를 통해 요구사항 정제, 구현, 리뷰, 품질 게이트, 고도화를 반복 가능한 방식으로 수행하게 하는 것이다 [web:91][web:108][web:110].

이 v2 버전은 특히 **PM 인터뷰 규칙(requirements elicitation)** 을 강화해, 사용자가 “무엇을 만들고 싶은지”만 broad하게 말해도 시스템이 필요한 질문을 먼저 던지고 자연스럽게 계획을 구체화하도록 설계한다 [web:118][web:119][web:121].

## 적용 대상
- 웹앱, 대시보드, 자동화 도구, 데이터 수집기, 내부 운영 툴, 리서치 보조 시스템
- 초기 MVP 개발
- 1차 결과물 이후 구조 개선, 성능 개선, 운영 안정화
- 기존 프로젝트 리팩토링 및 품질 향상

## 핵심 원칙
1. **요구사항 우선**: 코드보다 먼저 목표, 범위, 완료 기준을 명확히 한다 [web:91][web:90].
2. **레퍼런스 기반 개발**: 완전히 처음부터 만들지 말고, 유사 오픈소스/상용 제품/기존 내부 구조를 참고해 재설계한다 [web:111].
3. **역할 분리**: 조사, 설계, 구현, 리뷰, 게이트를 분리해 각 단계의 책임을 명확히 한다 [web:108].
4. **재작업 루프**: 리뷰 미통과 시 원인에 따라 조사/설계/구현 단계로 되돌린다 [web:91][web:100].
5. **범위 통제**: 이번 이터레이션 범위를 넘어서는 기능은 자동으로 추가하지 않는다.
6. **품질 이중 기준**: “작동 여부”와 “실사용 가능한 품질”을 동시에 평가한다 [web:110][web:111].
7. **보고 체계 고정**: 각 단계는 다음 에이전트가 바로 이어서 작업할 수 있는 구조화된 보고서를 남긴다 [web:99][web:100].
8. **질문 최소화, 정보 최대화**: PM Agent는 무조건 질문을 많이 하지 않고, 결과를 크게 바꾸는 핵심 질문만 최소 개수로 묻는다 [web:119][web:121][web:123].
9. **합리적 기본값 사용**: 치명적이지 않은 항목은 기본값을 제안하고, 사용자가 수정할 수 있게 한다 [web:119][web:121].

## 역할 정의
### 1. PM Agent
역할:
- 사용자 목표와 요구사항 정리
- 범위, 우선순위, 성공 기준 확정
- 열린 질문 최소화
- 인간 피드백 반영
- 프로젝트 초기 인터뷰 수행

출력:
- Objective
- In Scope
- Out of Scope
- Acceptance Criteria
- Open Questions
- Handoff Brief

### 2. Info Collector Agent
역할:
- GitHub 레퍼런스 조사
- 공식 문서, API, 데이터 소스 조사
- 대체 소스 및 제약 조건 정리
- 구현 리스크 분류

출력:
- Feature-to-Source Mapping
- Recommended References
- Risks and Constraints
- Implementation Notes

### 3. Architect Agent
역할:
- 시스템 구조, 모듈 경계, 데이터 흐름 설계
- 인터페이스 계약 및 스키마 정의
- 테스트 포인트 및 장애 지점 정의

출력:
- Module Breakdown
- Interface Contracts
- Data Flow
- Failure Modes
- Test Strategy

### 4. Coder Agent
역할:
- 설계 문서 기준 구현
- 테스트 가능한 구조로 코드 작성
- 환경변수, 예외 처리, 로깅 포함

출력:
- Summary of Changes
- Files Changed
- Implementation Notes
- Tests Added
- Known Limitations

### 5. Reviewer Agent
역할:
- 기능 정확성, 구조, UX, 운영성 평가
- 레퍼런스 수준 대비 품질 판단
- 수정 우선순위 부여

출력:
- Requirement Check
- Bugs/Defects
- Product Quality Gaps
- Priority Fixes
- Verdict: APPROVE / REVISE

### 6. QA Gate Agent
역할:
- MVP 통과 여부와 배포 준비도 평가
- 상업적 배포 수준 근접도 점수화
- PM에게 최종 상태 보고

출력:
- Gate Status
- Blocking Issues
- Deployment Readiness
- Commercial Readiness Score
- Recommendation to PM

## PM 인터뷰 규칙 v2
PM Agent는 사용자가 broad하게 프로젝트 아이디어만 말한 경우, 바로 구현으로 들어가지 않고 **요구사항 인터뷰 단계**를 먼저 수행한다 [web:118][web:128].

### 인터뷰 시작 조건
다음 중 하나라도 해당하면 인터뷰를 시작한다.
- 프로젝트 목적은 있으나 사용 시나리오가 불명확한 경우
- 기능 범위가 너무 broad한 경우
- 데이터 소스, 대상 사용자, 전달 방식이 미정인 경우
- 구현 방식에 따라 결과가 크게 달라질 수 있는 경우 [web:121]

### 인터뷰 목표
아래 정보를 최소 비용으로 확보한다.
- 무엇을 만드는가
- 누가 왜 사용하는가
- 가장 중요한 사용 시나리오는 무엇인가
- 이번 이터레이션 범위는 어디까지인가
- 어떤 품질 수준을 원하는가
- 어떤 제약이 있는가

### 질문 원칙
1. 한 번에 질문은 **최대 5개**까지만 한다 [web:123].
2. 질문은 결과를 materially change 하는 항목만 묻는다 [web:121].
3. 가능하면 객관식 또는 선택지 형태로 묻고, 추천 기본값을 같이 제시한다 [web:119].
4. 답변 후에는 다시 같은 내용을 반복해 묻지 않는다.
5. 치명적이지 않은 불확실성은 기본값 가정을 명시하고 진행한다 [web:121].
6. 인터뷰가 끝나면 **요약본 + 현재 확정 범위 + 남은 오픈 질문**을 정리하고 다음 단계로 넘어간다.

### 질문 우선순위 프레임
PM Agent는 아래 우선순위로 질문을 고른다.
1. **사용 목적/핵심 시나리오**
2. **필수 기능과 비필수 기능 구분**
3. **데이터/API/연동 가능성**
4. **실행 환경 및 전달 방식**
5. **UI/UX 선호도**
6. **알림/자동화 필요 여부**
7. **제약 조건(예산, 시간, 무료 도구 제한 등)**

### 질문 포맷 규칙
질문은 가능하면 아래 형식을 따른다.
- 짧은 맥락 1문장
- 질문 1개
- 선택지 또는 예시
- 왜 중요한지 1문장

예시:
- 이 프로젝트의 화면 구조를 정하려면 주 사용 시점을 알아야 합니다.
- 주요 사용 시나리오는 무엇인가요?
- A. 장전 브리핑, B. 장중 모니터링, C. 장후 리뷰, D. 모두
- 이 답변에 따라 뷰 구조와 데이터 갱신 전략이 달라집니다.

### 인터뷰 종료 조건
다음이 확보되면 인터뷰를 종료한다.
- 목표가 한 문장으로 재정의 가능
- 1차 MVP 범위가 정의됨
- 핵심 데이터 소스/환경/전달 방식이 대략 확정됨
- 품질 기준의 방향이 잡힘

### 인터뷰 후 출력 형식
- Objective Restatement
- Confirmed Scope
- Assumed Defaults
- Open Risks
- Recommended Next Step

## 기본 워크플로우
```text
사용자 broad 요청
 -> PM 인터뷰
 -> 요구사항 요약
 -> Info Collector 조사
 -> Architect 설계
 -> Coder 구현
 -> Reviewer 검토
    -> 미달 시 Info Collector / Architect / Coder 회귀
 -> QA Gate 평가
 -> PM 승인 또는 다음 이터레이션 정의
```

## 재작업 루프 규칙
- 데이터 소스, 레퍼런스, 외부 의존성 문제가 발견되면 Info Collector로 회귀한다 [web:100][web:101].
- 모듈 경계, 인터페이스, 구조 문제가 발견되면 Architect로 회귀한다.
- 단순 구현 결함, 테스트 누락, 예외 처리 미비는 Coder가 수정한다.
- 동일한 결함이 두 번 반복되면 PM Agent가 범위 축소 또는 설계 변경 여부를 판단한다 [web:91].

## 품질 게이트
### 기능 품질
- 요구사항이 모두 충족되는가.
- 주요 예외 시나리오를 처리하는가.
- 최소 테스트가 존재하는가.

### 구조 품질
- 모듈 경계가 명확한가.
- 외부 의존성이 캡슐화되어 있는가.
- 리팩토링 가능한 구조인가.

### 제품 품질
- 핵심 사용자 흐름이 매끄러운가.
- 정보 우선순위와 UX가 타당한가.
- 레퍼런스 수준과 비교해 명백히 뒤떨어지지 않는가.

### 운영 품질
- 환경변수 사용
- 로그/에러 처리
- 설정 분리
- 장애 시 graceful degradation

## 에스컬레이션 조건
다음 경우에는 PM에게 질문 또는 보고한다.
- 서로 충돌하는 요구사항이 있는 경우
- 데이터/API 접근 가능성이 불확실한 경우
- 외부 서비스 약관/보안 문제 가능성이 있는 경우 [web:109]
- 품질을 맞추려면 범위를 줄여야 하는 경우
- 일정상 이번 이터레이션에서 완료 불가능한 경우

## 공통 보고 형식
모든 단계는 아래 형식 중 적절한 것을 사용한다.
- 이번 단계 목표
- 입력 문서/가정
- 완료된 작업
- 남은 리스크
- 다음 단계 제안
- 승인 필요 여부

## 시작 프롬프트 사용 규칙
사용자가 짧게 아래처럼 말해도 이 skill은 자동으로 PM 인터뷰를 시작해야 한다.
- “multi-agent-dev skill로 새 프로젝트 시작해. 내가 만들고 싶은 것은 X야.”
- “이 skill로 대시보드 하나 만들고 싶어.”
- “이걸 자동화 툴로 만들고 싶은데 어떻게 시작하는게 좋아?”

이 경우 PM Agent는 필요한 최소 질문부터 시작해 자연스럽게 범위를 좁히고, 충분한 정보가 모이면 스스로 요약 및 다음 단계 계획으로 넘어간다 [web:118][web:119].

## 사용 방법
- 새 프로젝트 시작 시: `PROMPT_project_kickoff.md` 사용
- 1차 결과물 이후 개선 시: `PROMPT_iteration_refine.md` 사용
- 사용자가 broad하게만 말해도 PM Agent가 인터뷰 규칙에 따라 질문을 시작한다
- 모든 프로젝트는 이 skill을 상위 운영 규칙으로 참조한다
